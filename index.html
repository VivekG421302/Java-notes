1.1 How Java Works Under the Hood

Deep Dive: The Java Execution Pipeline

Imagine you are a chef writing a recipe in English. Your friend, who only speaks French, wants to cook the same dish. Instead of translating the recipe for every French-speaking friend, you write it once in a universal "food language" that any chef can understand. A local translator then converts it to their native tongue.

Java works the same way. Your source code (.java) is like the English recipe. The compiler (javac) translates it to bytecode (.class) - the universal food language. The JVM then translates bytecode to native machine code for whichever operating system (Windows, Linux, macOS) is running the program.

This "write once, run anywhere" philosophy is Java's superpower. The JVM is your universal translator.

Garbage Collection Algorithms

GC Process Flow

Garbage Collection follows a simple principle: find objects that are no longer reachable and reclaim their memory. Modern JVMs use generational collection based on the observation that most objects die young. The Heap is divided into Young Generation (Eden + 2 Survivor spaces) and Old Generation. New objects are allocated in Eden. When Eden fills, a minor GC moves survivors to a Survivor space. Objects that survive multiple GC cycles get promoted to Old Generation. When Old Generation fills, a more expensive major GC (Full GC) runs.

Just-In-Time (JIT) Compiler

The JIT compiler transforms Java from an "interpreted language" into a high-performance competitor to C++. As the JVM executes bytecode, the HotSpot Profiler monitors which methods are called frequently - these are called "hot methods." Once a method crosses an invocation threshold (typically 10,000 calls for C1, 15,000+ for C2), the JIT compiles the bytecode to highly optimized native machine code.

Analogy: The JIT is a Smart Translator

Imagine translating a speech in real-time. At first, you translate word-by-word (interpretation). But you notice the speaker keeps saying "machine learning model training" - so you prepare a pre-translated phrase and swap it in instantly. The JIT does exactly this: it starts interpreting, identifies hot code, then replaces it with optimized native code. Future calls skip interpretation entirely and run at native speed.

Production Gotcha: The Warm-Up Problem

Never measure performance immediately after JVM startup. Code starts in interpreted mode and only gets JIT-compiled after thousands of invocations. In production, always implement a warm-up period (send synthetic traffic for 30-60 seconds) before routing real traffic. Use -XX:+PrintCompilation (or unified logging) to see when methods get compiled. Also beware of deoptimization: if a method's assumptions fail (e.g., a usually-monomorphic call site suddenly receives a new type), the JVM falls back to interpreted mode, causing latency spikes.

// ConcurrentHashMap: lock-free reads, segmented writes
import java.util.concurrent.ConcurrentHashMap;

public class ConcurrentCache {
    // Java 8+: uses CAS + synchronized on first node per bucket
    // 16 segments by default, each bucket independently lockable
    private final ConcurrentHashMap<String, User> cache =
        new ConcurrentHashMap<>(1000);

    public User getUser(String id) {
        // computeIfAbsent: atomic check-then-act, no race conditions
        return cache.computeIfAbsent(id, this::loadFromDatabase);
    }

    public void incrementCounter(String key) {
        cache.merge(key, 1, Integer::sum);
    }

Java Generics

Deep Dive: Type Erasure

Java Generics use Type Erasure - a design decision made in 2004 for backward compatibility. At compile time, List<String> and List<Integer> are distinct types that catch errors. But after compilation, both become raw List with type information erased. The compiler inserts casts automatically. This is why you cannot do new T() or instanceof T at runtime - the JVM simply doesn't know what T was.

java 
// PECS Principle: Producer Extends, Consumer Super
// Use ? extends T when you only READ from the collection (Producer)
public void processAnimals(List<? extends Animal> animals) {
    // Can read: Animal a = animals.get(0);
    // Cannot write: animals.add(new Dog()); // COMPILE ERROR
    animals.forEach(Animal::makeSound);
}

// Use ? super T when you only WRITE to the collection (Consumer)
public void addDogs(List<? super Dog> list) {
    list.add(new Dog());
    list.add(new Puppy());
    // Can write, but reading only guarantees Object type
}

// Mnemonic: "PECS" = Producer-Extends, Consumer-Super

Exception Handling

java 
// Modern exception handling with try-with-resources (Java 7+)
public void processFile(String path) {
    // AutoCloseable resources closed in reverse order of opening
    try (BufferedReader reader = new BufferedReader(new FileReader(path));
         FileWriter writer = new FileWriter("output.txt")) {

        String line;
        while ((line = reader.readLine()) != null) {
            writer.write(line.toUpperCase() + "\n");
        }
    } catch (FileNotFoundException e) {
        // Specific: file doesn't exist
        throw new BusinessException("File not found: " + path, e);
    } catch (IOException e) {
        // General: any I/O error during read/write
        throw new BusinessException("I/O error processing file", e);
    }
}

Production Gotcha: Swallowing Exceptions

Never use empty catch blocks. The worst production bug is the silent failure. If you catch an exception, either (1) log it with context using SLF4J: log.error("Failed to process order {}", orderId, e);, (2) wrap and rethrow with a domain exception, or (3) use a recovery strategy. Also, never catch Throwable - catching OutOfMemoryError leaves the JVM in an undefined state. Catch specific exceptions, log with correlation IDs, and always preserve the stack trace when wrapping.

Wrapper Classes & Autoboxing

Gotcha: The Integer Cache Trap

Java caches Integer objects for values -128 to 127. This creates a classic interview trap:

Integer a = 127;   Integer b = 127;   // a == b  → true (cached)
Integer c = 128;   Integer d = 128;   // c == d  → false (new objects!)
Integer e = new Integer(127);         // e == a  → false (explicit new)

Always use .equals() for wrapper comparisons. The == operator compares references, not values. The cache range is configurable via -XX:AutoBoxCacheMax=<n>.

Analogy: The Toy Box with Numbered Slots

Imagine a toy store with a special display case that has pre-made toys numbered -128 to 127. When a kid asks for toy #127, the storekeeper just hands them the pre-made one from the case — super fast! But when someone asks for toy #128, the storekeeper has to build a brand new one from scratch. Both toys look identical, but they are different physical objects. If you check "is this the exact same toy?" (==), the answer is no for #128 but yes for #127. However, if you check "does this toy have the same number?" (.equals()), the answer is always yes. The store never caches toys with decimal numbers (float/double) because there are infinite possibilities!

Gotcha: String Literal vs new String()
String a = "hello";              // SCP reference (1 object total)
String b = "hello";              // Same SCP reference (a == b → true)

The librarian's catalog doesn't know about your private copy unless you register it (.intern()). The library used to keep its catalog in a tiny locked room (PermGen), but now it's in the main hall where everyone can reach it (Heap).

Gotcha: Modifying During Iteration

Never use for-each or standard for with index to remove elements from a collection — it causes ConcurrentModificationException or skips elements. Use Iterator.remove() or Java 8+ removeIf():

// WRONG: skips elements, causes CME
for (String s : list) { if (s.startsWith("tmp")) list.remove(s); }

// CORRECT: Iterator
Iterator<String> it = list.iterator();
while (it.hasNext()) { if (it.next().startsWith("tmp")) it.remove(); }

// MODERN: Java 8+ removeIf
list.removeIf(s -> s.startsWith("tmp"));
Analogy: The School Bus Route

A for-each loop is like a bus that visits every house on a street in order, but you're not allowed to knock down houses while riding! An Iterator is like having a demolition permit — you can remove a house, but only one at a time, and only through the proper city office (it.remove()).

            if (conn != null) {
                try { conn.close(); } catch (SQLException ignored) {}
            }
        }
    }

    @Override
    protected void finalize() throws Throwable {  // DEPRECATED — DO NOT USE
        super.finalize();  // Unreliable, single-threaded, can resurrect objects

Composition Over Inheritance

Deep Dive: Why Composition Wins

Inheritance creates a rigid "is-a" relationship that breaks when requirements change. If you model Employee extends Person, what happens when an Employee also needs to be a Customer? As the Gang of Four stated: "Favor object composition over class inheritance."

Production Gotcha: The Fragile Base Class Problem

When you inherit from a class, you are tightly coupled to its implementation details. If the parent class changes a method you override, your subclass may break silently. In Java, always use @Override to catch signature mismatches. Deep Dive: Singleton Pattern — From Naive to Correct

1. Naive (NOT thread-safe)

public class NaiveSingleton {
    private static NaiveSingleton instance;
    private NaiveSingleton() {}
    public static NaiveSingleton getInstance() {
        if (instance == null) instance = new NaiveSingleton();  // RACE CONDITION!
        return instance;
    }
}

2. Double-Checked Locking (CORRECT)

public class SafeSingleton {
    private static volatile SafeSingleton instance;  // volatile is CRITICAL
    private SafeSingleton() {}
    public static SafeSingleton getInstance() {
        if (instance == null) {                          // first check (no lock)
            synchronized (SafeSingleton.class) {
                if (instance == null) {                  // second check (with lock)
                    instance = new SafeSingleton();
                }
            }
        }
        return instance;
    }
}

volatile prevents instruction reordering: without it, another thread could see a partially constructed object.

Gotcha: Never Call Overridable Methods from Constructors
class Parent {
    Parent() { initialize(); }  // DANGEROUS!

}

class Child extends Parent {

    @Override
    void initialize() {

2.1 Concurrency & Multithreading

Deep Dive: Why Concurrency Matters

Modern CPUs have stopped getting faster in terms of clock speed. Instead, they gain more cores - 8, 16, 32, even 64 cores are common in servers. If your program uses only one thread, you are leaving 95% of your CPU's power unused. Concurrency is how we unlock that power: running multiple tasks simultaneously, either by switching rapidly between them on a single core (multitasking) or running them truly in parallel on multiple cores.

But concurrency introduces complexity. When two threads access the same data simultaneously, race conditions occur. When threads wait for each other, deadlocks happen. The Java Concurrency Utilities provide battle-tested tools to manage this complexity safely.

public class VirtualThreadServer {

    // 1. Create virtual thread per task (structured concurrency)
    public void handleRequests(List<Request> requests) {
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            List<StructuredTaskScope.Subtask<Response>> subtasks =
                requests.stream()
                    .map(req -> scope.fork(() -> process(req)))
                    .toList();

            scope.join();           // Wait for all to complete
            scope.throwIfFailed();  // If any failed, propagate

            return subtasks.stream()
                .map(StructuredTaskScope.Subtask::get)
                .toList();
        }
    }

    // spring.threads.virtual.enabled=true

Production Gotcha: Thread Pinning

Virtual threads can be pinned to platform threads when executing synchronized blocks or native methods. This negates their benefits - the carrier thread is blocked and cannot mount other virtual threads. In Java 21+, use java.util.concurrent.locks.ReentrantLock instead of synchronized inside virtual threads. Monitor with -Djdk.tracePinnedThreads=full. Also, avoid thread-local variables with virtual threads - they don't work as expected since a virtual thread may run on different carriers over time. Use ScopedValue (Java 21+) instead.

Deep Dive: What is a Servlet?

A Servlet is Java's fundamental building block for web applications. Think of it as a "Java program that speaks HTTP." When a user types a URL in their browser, the HTTP request travels through the internet to a Servlet Container (like Tomcat or Jetty). The container looks up which Servlet handles that URL, creates a thread, and calls the Servlet's service() method. The Servlet reads the request, performs business logic, and writes an HTTP response. This request-response cycle is the foundation of every Java web framework, including Spring MVC.

Production Gotcha: The N+1 Select Problem

The N+1 problem is the most common Hibernate performance killer. When you load N entities and lazily access a relationship, Hibernate executes 1 initial query + N additional queries. For 100 orders with customer relationships, that's 101 queries instead of 1. Solutions: (1) Use JOIN FETCH in JPQL: SELECT o FROM Order o JOIN FETCH o.customer, (2) Use @EntityGraph to declare fetch plans, (3) Enable batch fetching: @BatchSize(size = 50), (4) Use a DTO projection for read-only queries. Always check SQL logs in development - if you see a query repeating in a loop, you have N+1.

Aspect-Oriented Programming (AOP)

Analogy: AOP is Like Airport Security

Imagine you run an airport with 50 gates. You need to add security checks to every gate. You could modify each gate's code (cross-cutting concern) - but that's error-prone. Instead, you install security checkpoints that intercept passengers before they reach any gate. AOP works the same way: you define Advices (security checks) at Pointcuts (which gates) without modifying the actual gate code. Logging, transaction management, and security are classic cross-cutting concerns.

java 
// Centralized logging aspect
@Aspect
@Component
public class LoggingAspect {

    // Pointcut: matches all methods in service package
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceMethods() {}

    @Around("serviceMethods()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        log.info("Entering: {}", joinPoint.getSignature().getName());

        try {
            Object result = joinPoint.proceed();  // Execute the actual method
            long duration = System.currentTimeMillis() - start;
            log.info("Exiting: {} ({}ms)", joinPoint.getSignature().getName(), duration);
            return result;
        } catch (Exception e) {
            log.error("Exception in {}: {}", joinPoint.getSignature(), e.getMessage());
            throw e;
        }
    }
}

// Transaction aspect (Spring's @Transactional is implemented this way)
@Aspect
@Component
public class TransactionAspect {
    @Around("@annotation(transactional)")
    public Object manageTransaction(ProceedingJoinPoint jp, Transactional transactional)
            throws Throwable {
        TransactionStatus status = txManager.getTransaction(
            new DefaultTransactionDefinition());
        try {
            Object result = jp.proceed();
            txManager.commit(status);
            return result;
        } catch (RuntimeException e) {
            txManager.rollback(status);  // Rollback on unchecked exceptions
            throw e;
        }
    }
}

@Transactional Propagation: The Transaction Matrix

Propagation defines what happens when a @Transactional method calls another @Transactional method. Seven levels, seven behaviors.

PropagationExisting TX?BehaviorWhen to Use
REQUIRED (default)YesJoin existing TXAlmost everything
REQUIREDNoCreate new TX—
REQUIRES_NEWYesSuspend existing, create NEWAudit logs, notifications
REQUIRES_NEWNoCreate new TX—
NESTEDYesCreate savepointPartial rollback scenarios
SUPPORTSYesJoin existing TXRead-only queries
SUPPORTSNoExecute non-transactionallyOptional transactional behavior
MANDATORYYesJoin existing TXInternal service methods
MANDATORYNoThrow exceptionEnforce caller has TX
NOT_SUPPORTEDYesSuspend TX, run withoutReporting queries
NEVERYesThrow exceptionCache operations
Java
@Service
public class OrderService {
    @Autowired private PaymentService paymentService;
    @Autowired private AuditService auditService;
    @Autowired private OrderRepository orderRepo;

    @Transactional
    public void placeOrder(OrderRequest req) {
        // In TX #1
        Order order = orderRepo.save(new Order(req));

        // paymentService uses REQUIRED — joins TX #1
        // If payment fails, order is ALSO rolled back
        paymentService.processPayment(order.getId(), req.getAmount());

        // auditService uses REQUIRES_NEW — creates TX #2
        // Even if TX #1 rolls back, audit record is COMMITTED
        auditService.logOrderCreated(order.getId());
    }
}

@Service
public class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logOrderCreated(Long orderId) {
        auditRepo.save(new AuditLog("ORDER_CREATED", orderId));
    }
}

@Service
public class ReportService {
    // Run without TX — better for long-running read queries
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public List<OrderReport> generateMonthlyReport(YearMonth month) {
        return orderRepo.findByMonth(month); // No TX overhead
    }
}
Production Gotcha: REQUIRES_NEW creates a PHYSICAL database connection — it doesn't reuse the same connection. In high-concurrency systems, this can exhaust your connection pool. Also, NESTED (savepoints) only works with JDBC 3.0+ drivers and is NOT supported by all JPA providers (Hibernate supports it, EclipseLink doesn't). Test your specific database + JPA combo.
Analogy for a 10-Year-Old
Imagine a bank with safety deposit boxes (transactions):

REQUIRED = "Use my box if there's one open, otherwise get a new one." You and your friend share the same box. If either of you messes up, BOTH items are removed.

REQUIRES_NEW = "I ALWAYS need my own private box." Even if you already have a box, you PAUSE using it, open a NEW one, do your thing, close it, then resume your original box. Your new box is independent.

NESTED = "Put a divider in my current box." If the section behind the divider goes wrong, only THAT section is thrown away. The rest of the box stays.

MANDATORY = "I refuse to work unless you already have a box open." If you try to use this service without a box, they kick you out (throw exception).

NEVER = "I refuse to work if there's ANY box open." Perfect for things that shouldn't interfere with banking — like checking the weather.

@EnableConfigurationProperties(HttpClientProperties.class)

            .setReadTimeout(props.getReadTimeout())

Retrieval-Augmented Generation (RAG)

Analogy: RAG is Like an Open-Book Exam

Imagine taking a test about your company's policies. Without RAG, the AI is like a student taking a closed-book exam about a company they don't work for - it will guess or hallucinate. With RAG, you first load all company documents into a Vector Database (like a searchable index). When a user asks a question, you search for the most relevant document chunks, add them to the prompt as context, and then ask the AI to answer. The AI now has the exact information it needs - like an open-book exam with a perfectly organized reference library.

java 
// RAG Implementation: Question Answering over private documents
@Service
public class RagService {

    private final VectorStore vectorStore;
    private final ChatClient chatClient;
    private final EmbeddingModel embeddingModel;

    // Step 1: Ingest documents into vector store
    public void ingestDocuments(List<Document> documents) {
        // Documents are split into chunks, embedded, and stored
        TokenTextSplitter splitter = new TokenTextSplitter(500, 50);
        List<Document> chunks = splitter.apply(documents);
        vectorStore.add(chunks);  // Pgvector, Pinecone, Chroma, etc.
    }

    // Step 2: Answer questions using retrieved context
    public String answerQuestion(String question) {
        // 2a. Convert question to embedding vector
        // 2b. Find top-5 most similar document chunks
        List<Document> relevantDocs = vectorStore.similaritySearch(
            SearchRequest.query(question).withTopK(5));

        // 2c. Augment prompt with retrieved context
        String context = relevantDocs.stream()
            .map(Document::getContent)
            .collect(Collectors.joining("\n\n---\n\n"));

        // 2d. Generate answer grounded in retrieved context
        return chatClient.prompt()
            .system("Answer based ONLY on the provided context. " +
                     "If you cannot answer from the context, say 'I don't know.'")
            .user(u -> u.text("Context:\n{context}\n\nQuestion: {question}")
                .param("context", context)
                .param("question", question))
            .call()
            .content();
    }
}

5.1 Spring Security & Stateless Authentication

Deep Dive: Token-Based vs Session-Based Authentication

Traditional session-based authentication stores user state on the server. When you log in, the server creates a session (stored in memory or Redis) and sends you a cookie with a session ID. Every subsequent request includes this cookie, and the server looks up your session. This works but has problems: sessions consume server memory, don't scale horizontally without sticky sessions or shared Redis, and are vulnerable to CSRF attacks.

Token-based (JWT) authentication is stateless. The server stores nothing. After login, the server issues a signed JWT token containing user claims (username, roles, expiry). The client sends this token in every request header. The server validates the signature and extracts claims - no database lookup needed for authentication. This scales infinitely and works naturally with microservices and SPAs.

@Component
public class JwtUtil {

    private final SecretKey key = Jwts.SIG.HS256.key().build();
    private final long expirationMs = 86400000; // 24 hours

    public String generateToken(UserDetails user) {
        return Jwts.builder()
            .subject(user.getUsername())
            .claim("roles", user.getAuthorities().stream()
                .map(GrantedAuthority::getAuthority)
                .toList())
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expirationMs))
            .signWith(key)
            .compact();
    }

    public Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(key)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    public boolean isValid(String token) {
        try {
            parseToken(token);
            return true;
        } catch (JwtException | IllegalArgumentException e) {
            return false;
        }
    }
}

Production Gotcha: JWT Security Checklist

Never store JWTs in localStorage - vulnerable to XSS attacks. Use httpOnly cookies instead. Always validate all JWT claims: check exp (expiration), iss (issuer), aud (audience), and verify the signature algorithm isn't "none". Store the signing key in a secrets manager (AWS Secrets Manager, HashiCorp Vault), never in source code. Implement token refresh: short-lived access tokens (15 min) + long-lived refresh tokens (7 days) stored securely. For logout, maintain a token blocklist in Redis (with TTL matching token expiry). Rotate signing keys periodically using kid (key ID) header to identify which key signed the token.

// Resilience4j annotations for microservice fault tolerance
@Service
public class PaymentGatewayClient {

    // Circuit Breaker: Open after 50% failure rate, wait 30s before retry
    @CircuitBreaker(name = "payment", fallbackMethod = "processPaymentFallback")
    @Retry(name = "payment")       // Retry 3 times with exponential backoff
    @TimeLimiter(name = "payment")   // Timeout after 5 seconds
    public CompletableFuture<PaymentResult> processPayment(PaymentRequest request) {
        return CompletableFuture.supplyAsync(() -> {
            // Call external payment gateway
            return restClient.post()
                .uri("/api/v1/payments")
                .body(request)
                .retrieve()
                .body(PaymentResult.class);
        });
    }

    // Fallback: called when circuit is open or all retries exhausted
    private CompletableFuture<PaymentResult> processPaymentFallback(
            PaymentRequest request, Exception ex) {
        log.warn("Payment failed, queueing for retry: {}", request.getId());
        // Queue to dead-letter queue for later processing
        messageQueue.send("payment-dlq", request);
        return CompletableFuture.completedFuture(
            PaymentResult.queued(request.getId()));
    }
}

Production Gotcha: Circuit Breaker Misconfiguration

The most common Circuit Breaker mistake is setting the sliding window too small. With slidingWindowSize: 5, a single failure gives 20% failure rate - too sensitive for transient network blips. Use at least 100 for production. Also, always provide fallback methods - without them, the CB throws CallNotPermittedException to the caller. Never combine CB with synchronous HTTP calls inside a loop - the CB state is per-instance, so it may not trigger correctly. Use @TimeLimiter with CompletableFuture to prevent hanging threads. Monitor CB metrics via Actuator + Micrometer for early warning.

Deep Dive: Why CI/CD Matters

Manual deployments are the single biggest source of production incidents. "It worked on my machine" is not a deployment strategy. 8.1 Advanced SQL Query Mechanics & Execution Plans

Deep Dive: How MySQL Executes a Query

When you send a SQL query to MySQL, it doesn't just read tables sequentially. The query goes through multiple stages: parsing (check syntax), preprocessing (validate columns/tables), optimization (find the cheapest execution plan), and execution. The optimizer considers multiple strategies: which index to use, which table to read first in a JOIN, whether to use a temporary table or sort in memory. Understanding how to inspect and influence these decisions separates database administrators from developers who wonder why their "simple" query takes 30 seconds.

-- Key columns in EXPLAIN output:
-- type: ALL (full table scan), index, range, ref, eq_ref, const
--       ALL is BAD - means reading every row
--       const/eq_ref is GOOD - single row lookup
--
-- possible_keys: Which indexes the optimizer considered
-- key: Which index was actually chosen (NULL = no index)
-- rows: Estimated rows to examine (lower is better)
-- filtered: % of rows that match the WHERE condition
-- Extra: Additional info - "Using filesort" or "Using temporary" are warning signs
EXPLAIN Output Interpretation Guide

type column (most important):

  • const / system: Best case - single row lookup by primary key
  • eq_ref: One row read per row from previous table (JOIN on PK/UK)
  • ref: Index lookup returning potentially multiple rows
  • range: Index range scan (WHERE id BETWEEN 10 AND 100)
  • index: Full index scan (better than ALL, still not ideal)
  • ALL: Worst case - full table scan. Every row examined.

Extra column (warning signs):

  • Using filesort: MySQL must sort after retrieving rows - add an index
  • Using temporary: MySQL creates a temp table - simplify GROUP BY/ORDER BY
  • Using where: Filtering after index lookup - may need better index

Indexing Deep Dive

Analogy: Indexes are Like Book Indexes

Imagine a 1000-page textbook without an index. To find "garbage collection," you read every page. With an index at the back, you jump directly to page 427. Database indexes work identically: a separate data structure (B+Tree) maps key values to row locations. The trade-off: indexes speed up reads but slow down writes (INSERT, UPDATE, DELETE must maintain the index). Every index adds storage overhead and slows down write operations. A table with 10 indexes might read fast but write 10x slower.

sql 
-- InnoDB B+Tree Index Structure
-- Clustered Index (Primary Key):
--   Leaf nodes contain the ACTUAL row data
--   This is why PRIMARY KEY lookups are so fast
--   Only ONE clustered index per table

-- Secondary Index (Non-Clustered):
--   Leaf nodes contain: [index columns] + [primary key value]
--   If query needs columns not in index, must do "lookup" to clustered index

-- Creating optimal indexes
-- 1. Primary key (auto-created as clustered index)
ALTER TABLE orders ADD PRIMARY KEY (id);

-- 2. Composite index with Leftmost Prefix Rule
-- If you query by (country, city, zip), create index in that order
ALTER TABLE customers ADD INDEX idx_geo (country, city, zip);
-- This index supports: country / country+city / country+city+zip
-- It does NOT support: city alone / zip alone (leftmost prefix rule!)

-- 3. Covering Index: includes ALL columns needed by query
-- The query can be satisfied entirely from the index, no table lookup
ALTER TABLE orders ADD INDEX idx_covering (customer_id, status, created_at, total);
-- Query: SELECT total FROM orders WHERE customer_id = ? AND status = ?
-- This is covered! No need to access the actual table data.

-- 4. Check index usage (MySQL 8.0+)
SELECT INDEX_NAME, CARDINALITY
FROM information_schema.STATISTICS
WHERE TABLE_NAME = 'orders';
-- CARDINALITY / total_rows ratio should be close to 1 for unique indexes

Production Gotcha: Index Abuse

Creating indexes on every column is a common anti-pattern. Each index adds ~20% overhead to INSERT/UPDATE/DELETE operations and consumes disk space. Monitor with sys.schema_unused_indexes to find indexes that have never been used. Avoid redundant indexes: INDEX(a) is redundant if you have INDEX(a, b). Watch for duplicate indexes created by different developers. For tables with heavy write loads (logs, events), prefer write-optimized engines or batch inserts. Use ALTER TABLE ... ALGORITHM=INPLACE for online index creation to avoid locking the table in production.

Join Execution Algorithms

sql 
-- Nested Loop Join: For each row in outer table, scan inner table
-- Best when: outer table is small, inner table has index on join column
-- Cost: O(outer_rows * inner_rows) without index, O(outer_rows * log(inner_rows)) with index

-- Hash Join (MySQL 8.0.18+): Build hash table from smaller table, probe with larger
-- Best when: no suitable index, large tables

-- Optimizing JOINs: Always put the most restrictive table first
-- (MySQL optimizer usually handles this, but explicit helps readability)

-- BAD: Joining before filtering (large intermediate result)
SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.id  -- 1M orders joined first
WHERE c.country = 'USA';                    -- Then filtered

-- GOOD: Filter first, then join (smaller working set)
SELECT *
FROM customers c
JOIN orders o ON o.customer_id = c.id
WHERE c.country = 'USA';  -- Filter ~500 USA customers first, then join their orders

Stored Procedures & Database Programming

sql 
-- Stored Procedure: Encapsulate complex business logic in database
DELIMITER //
CREATE PROCEDURE ProcessMonthlyPayroll(
    IN payroll_month DATE,
    OUT total_processed INT,
    OUT total_amount DECIMAL(12,2)
)
BEGIN
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        -- Automatic rollback on any error
        ROLLBACK;
        RESIGNAL;
    END;

    START TRANSACTION;

    -- Calculate payroll for all active employees
    INSERT INTO payroll_entries (employee_id, month, base_salary, bonus, total)
    SELECT
        e.id,
        payroll_month,
        e.base_salary,
        COALESCE(SUM(b.amount), 0),
        e.base_salary + COALESCE(SUM(b.amount), 0)
    FROM employees e
    LEFT JOIN bonuses b ON b.employee_id = e.id
        AND b.bonus_month = payroll_month
    WHERE e.status = 'ACTIVE'
    GROUP BY e.id, e.base_salary;

    SET total_processed = ROW_COUNT();

    SELECT SUM(total) INTO total_amount
    FROM payroll_entries
    WHERE month = payroll_month;

    COMMIT;
END//
DELIMITER ;
Analogy: Stored Procedures are Like a Vending Machine

A stored procedure is like a vending machine: you put in money (parameters) and get a snack (result) - you don't need to know the internal mechanics. The benefits: reduced network round-trips (one CALL vs. multiple queries), centralized business logic, and security (grant EXECUTE instead of SELECT/INSERT). The trade-offs: harder to version control (not in Git by default), database-specific (not portable), and harder to unit test. Modern Spring Boot applications typically prefer keeping business logic in Java, using stored procedures only for heavy data processing or legacy integration.

Production Gotcha: Deadlocks in High-Concurrency Systems

Deadlocks occur when two transactions each hold a lock the other needs. MySQL detects and kills one (rollback), but your application must handle the retry. Prevention strategies: (1) Always acquire locks in the same order across all transactions, (2) Keep transactions as short as possible - never call external APIs inside a transaction, (3) Use SELECT ... FOR UPDATE NOWAIT to fail fast instead of waiting, (4) Set innodb_lock_wait_timeout to a reasonable value (not the default 50s). For Spring: use @Transactional(timeout = 5) and implement retry with exponential backoff using Spring Retry. Monitor SHOW ENGINE INNODB STATUS for deadlock analysis.

String Constant Pool: From PermGen to Heap
Historical Fact
Before Java 7, the String Constant Pool lived in the PermGen (Permanent Generation) space, a fixed-size memory area separate from the main heap. This caused OutOfMemoryError: PermGen space in applications that heavily used String.intern().
The Java 7 Revolution
In Java 7, the String Constant Pool was moved to the main Java heap (young and old generations). This change eliminated PermGen OOMs from string interning but required larger heap sizing. The -XX:StringTableSize flag controls the number of buckets (default 65,536 since Java 11).
Java 8: PermGen Eliminated
Java 8 completely removed PermGen, replacing it with Metaspace (native memory, auto-growing). Class metadata now lives in Metaspace, while interned strings remain in the heap. The -XX:MaxPermSize flag no longer exists.
Q: What does new String("hello").intern() do in Java 8+?
A: It creates a String object in the heap via new, then adds "hello" to the String Constant Pool (also in the heap) if not present. The intern() method returns the SCP reference. Two objects exist temporarily; the heap copy becomes eligible for GC if not referenced.
Q: Why did moving the SCP to the heap improve GC?
A: Because SCP strings are now subject to normal garbage collection. In PermGen, they were effectively immortal (only collected during full GCs). In the heap, unused interned strings in the young generation are collected during minor GCs, reducing memory pressure.
finalize() Deprecation: Why and What Replaces It
Deprecation Timeline
Object.finalize() was deprecated in Java 9 (2017) and marked @Deprecated(forRemoval=true) in Java 18 via JEP 421. It has never worked reliably: the JVM guarantees neither timely execution nor execution at all. Finalizers can cause object resurrection, severe performance degradation, and resource leaks.
Why finalize() Failed
Finalizers run on a single system thread, creating a bottleneck. They can resurrect objects (making them reachable again), delaying GC indefinitely. There's no guarantee when or if cleanup occurs — native resources like file handles may remain open for arbitrary periods.
Modern Alternative: Cleaner API (Java 9+)
The java.lang.ref.Cleaner API provides explicit, controllable cleanup without object resurrection. Unlike finalizers, cleaners don't hold references to the object being cleaned, preventing resurrection bugs. Example: Cleaner.create().register(obj, cleanupAction).
Best Practice: try-with-resources
For resources with a defined lifecycle, implement AutoCloseable and use try-with-resources. This guarantees immediate, deterministic cleanup: try (Resource r = new Resource()) { ... }. The resource is closed automatically, even if exceptions occur.
Q: What happens if I still use finalize() in Java 18+?
A: The method still exists but generates compiler warnings. You can test removal impact with --finalization=disabled. The JDK team plans complete removal in a future release. Migrate to AutoCloseable + try-with-resources or the Cleaner API immediately.
Q: What's the difference between Cleaner and PhantomReference?
A: Both allow post-mortem cleanup. Cleaner is higher-level and easier to use — you register a Runnable that runs when the object becomes phantom reachable. PhantomReference is lower-level: you manually poll a ReferenceQueue and perform cleanup yourself. Cleaner is built on top of PhantomReference internally.
Thread Lifecycle: JVM Internals & OS Mapping
OS-Level Mapping
Every Java thread maps 1:1 to an OS thread (platform thread). The JVM calls pthread_create on Linux or CreateThread on Windows. Thread stacks default to 1MB on 64-bit JVMs. Virtual threads (Java 21+) break this 1:1 mapping — they are JVM-managed and multiplex onto a small pool of carrier threads.
State Transitions
NEW → RUNNABLE happens on start(). RUNNABLE → WAITING on Object.wait() (no timeout). RUNNABLE → TIMED_WAITING on Thread.sleep() or wait(timeout). RUNNABLE → BLOCKED when trying to enter a synchronized block held by another thread. TERMINATED is irreversible — the thread object remains in memory but can never be restarted.
Q: Can a thread go from WAITING directly to RUNNING?
A: No. A thread in WAITING must first transition to RUNNABLE via notify() or notifyAll(), then the OS scheduler decides when it actually gets CPU time. There's no direct WAITING → RUNNING path — the RUNNABLE state is the mandatory waiting room.
Q: Why does Thread.State have RUNNABLE but not RUNNING?
A: The JVM abstracts OS scheduling. At the OS level, a thread is either "ready to run" (in the run queue) or "currently executing on a CPU core." The JVM collapses both into RUNNABLE because it cannot reliably distinguish — the OS can preempt a thread at any moment, making the distinction meaningless from Java's perspective.
Q: What happens if you call start() twice on the same Thread?
A: IllegalThreadStateException is thrown. The thread's internal state machine only allows start() when in the NEW state. Once moved to RUNNABLE, the transition is irreversible. This is enforced by a boolean started flag in the Thread class.
synchronized vs ReentrantLock vs volatile
Monitor vs Lock
synchronized uses the object's monitor (intrinsic lock) — every Java object has a hidden monitor. ReentrantLock is a separate object with its own state. Monitors are simpler but less flexible: no timeout, no interruptible acquisition, no fairness option, no multiple Condition variables. ReentrantLock offers all of these but requires explicit unlock() in a finally block.
volatile Memory Semantics
volatile guarantees: (1) visibility — writes are immediately visible to other threads, and (2) happens-before ordering — reads cannot be reordered before writes. But volatile does NOT guarantee atomicity. volatile++ is NOT thread-safe because it's read-modify-write. Use AtomicInteger or synchronized for compound operations.
Q: When should I use synchronized over ReentrantLock?
A: Use synchronized when you need simple mutual exclusion and the block is short. It's less error-prone (no forgotten unlock), automatically released on exception, and the JVM can optimize it better (biased locking, lock elision). Use ReentrantLock when you need: timeouts, interruptible locks, fairness, multiple conditions, or try-lock semantics.
Q: Does volatile prevent all reordering?
A: No. volatile prevents reordering of the volatile variable's reads/writes relative to each other, but normal (non-volatile) reads/writes can still be reordered around them by the compiler and CPU. For full protection, use synchronized, Atomic classes, or the java.util.concurrent utilities.
Q: What is lock coarsening and lock elision?
A: Lock coarsening: The JIT merges adjacent synchronized blocks on the same object into one larger block, reducing lock/unlock overhead. Lock elision: The JIT detects that a lock is only ever acquired by one thread (no contention) and removes the lock entirely — the object is thread-local in practice. Both are JVM optimizations that make synchronized faster than it looks in microbenchmarks.
Thread Pool Tuning: Production Rules
Pool Sizing Formula
For CPU-bound tasks: poolSize = CPU cores + 1. For I/O-bound tasks: poolSize = CPU cores / (1 - blockingFactor), where blockingFactor is the fraction of time threads spend waiting. A thread waiting 50% of the time on a 4-core machine needs ~8 threads. Virtual threads (Java 21+) make this calculation irrelevant for I/O — use millions.
WorkQueue Choices
LinkedBlockingQueue: unbounded by default — can cause OOM if producers outpace consumers. ArrayBlockingQueue: bounded, fair (optional), less memory overhead. SynchronousQueue: no capacity — each insert must wait for a remove. Used in newCachedThreadPool. PriorityBlockingQueue: tasks ordered by priority. Choose based on backpressure needs.
Q: Why does newFixedThreadPool use an unbounded queue?
A: Executors.newFixedThreadPool() uses LinkedBlockingQueue with capacity Integer.MAX_VALUE. This means tasks never get rejected — they just pile up in memory. In production, this is dangerous: a traffic spike can cause OOM. Always use ThreadPoolExecutor directly with a bounded queue and a RejectedExecutionHandler (CallerRunsPolicy, DiscardPolicy, or custom).
Q: What's the difference between shutdown() and shutdownNow()?
A: shutdown() initiates an orderly shutdown: no new tasks accepted, but queued and running tasks complete. shutdownNow() attempts to stop all actively executing tasks, returns a list of tasks that were awaiting execution, and prevents new tasks. shutdownNow() sends Thread.interrupt() to running threads — tasks that don't respond to interruption continue running. Always follow with awaitTermination().
Q: How does ForkJoinPool differ from ThreadPoolExecutor?
A: ForkJoinPool uses work-stealing: each thread has its own deque. When a thread finishes its tasks, it steals from the tail of another thread's deque. This is optimal for divide-and-conquer algorithms where tasks spawn subtasks. ThreadPoolExecutor uses a shared queue — better for independent tasks. parallelStream() uses the common ForkJoinPool by default.
CompletableFuture: Async Pipeline Patterns
Default Executor
If you don't specify an Executor, CompletableFuture uses ForkJoinPool.commonPool() (size = CPU cores - 1). For I/O-bound tasks, this is too small. Always pass a custom Executor for async operations: CompletableFuture.supplyAsync(() -> ..., executor). Use a dedicated thread pool with meaningful names for observability.
Exception Propagation
In a chain like supplyAsync().thenApply().thenAccept(), an exception in any stage propagates to the final stage. Use exceptionally() for recovery at specific stages, handle() for unified success/failure handling, and whenComplete() for side effects (logging) without changing the result. join() throws unchecked CompletionException; get() throws checked ExecutionException.
Q: What's the difference between thenApply and thenCompose?
A: thenApply(f) applies a function to the result and returns a new CompletableFuture wrapping the function's return value. thenCompose(f) applies a function that returns a CompletableFuture and FLATTENS it — the result is the inner future's result, not a future-of-a-future. Use thenCompose when the next step is itself async (e.g., another API call).
Q: How do you implement a timeout in CompletableFuture?
A: Java 9+ provides orTimeout(timeout, unit) which completes exceptionally with TimeoutException after the delay. completeOnTimeout(defaultValue, timeout, unit) completes with a default value instead. For Java 8, use CompletableFuture.anyOf(cf, timeoutCf) where timeoutCf is created via a ScheduledExecutorService. Always set timeouts on external calls to prevent resource leaks.
Q: Can CompletableFuture chains cause memory leaks?
A: Yes. Long chains of thenApply/thenCompose without terminal operations (like get() or join()) can accumulate stages in memory. In a loop that creates infinite chains, each stage holds a reference to the previous one, forming a linked list. Always ensure chains terminate, and for hot loops, consider breaking the chain or using CompletableFuture.allOf() with a bounded batch size.
Virtual Threads: Production Tuning & Pinning
Pinning: The Silent Killer
Virtual threads are mounted on carrier (platform) threads. When a virtual thread blocks on I/O, it unmounts, freeing the carrier. But if the virtual thread holds a monitor (enters synchronized or calls native code), it PINNED to the carrier — the carrier cannot serve other virtual threads. This negates the benefit. Use jcmd <pid> Thread.dump_to_file or -Djdk.tracePinnedThreads=full to detect pinning in development.
Spring Boot 3.2+ Configuration
Enable with spring.threads.virtual.enabled=true. This configures Tomcat to use Executors.newVirtualThreadPerTaskExecutor() for request handling. For async operations, define a @Bean returning Executors.newVirtualThreadPerTaskExecutor(). For scheduled tasks, use @Scheduled with a virtual thread executor. Monitor carrier thread count (should be ~CPU cores) vs. virtual thread count (can be millions).
Q: Should I replace all synchronized with ReentrantLock for virtual threads?
A: Only if profiling shows pinning is a problem. synchronized is still fine for short, uncontended critical sections. The JVM is improving (JEP 491 in Java 24) to make synchronized virtual-thread-friendly. For now: measure first, optimize second. Use ReentrantLock for long-held locks or when you need its features (timeouts, fairness, conditions).
Q: How do ThreadLocals work with virtual threads?
A: ThreadLocal works but is expensive — each virtual thread gets its own copy, and with millions of virtual threads, memory usage balloons. Java 21 introduces Scoped Values (JEP 446) as a replacement: immutable, inheritable, and automatically cleaned up when the scope ends. For new code, prefer Scoped Values. For legacy code, audit ThreadLocal usage and consider migration.
Q: What's the recommended migration strategy to virtual threads?
A: (1) Enable on a single non-critical endpoint with a feature flag. (2) Run load tests and check for pinning with -Djdk.tracePinnedThreads. (3) Fix pinning by replacing synchronized with ReentrantLock in hot paths. (4) Gradually expand to more endpoints. (5) Monitor carrier thread saturation — if carriers are always busy, you may have CPU-bound work that needs platform threads. Never enable globally without testing.
Parallel Streams: Pitfalls & Best Practices
The ForkJoinPool Bottleneck
All parallel streams share the ForkJoinPool.commonPool(), which has Runtime.getRuntime().availableProcessors() - 1 threads. A single misbehaving parallel stream can starve all others. Use list.parallelStream() cautiously in shared code. For custom parallelism, submit the stream operation to your own ForkJoinPool: pool.submit(() -> list.parallelStream().map(...).collect(...)).get().
Stateful Operations Are Dangerous
sorted(), distinct(), and limit() are stateful — they need to see all elements. In parallel streams, this requires coordination (merging partial results), which can be expensive. sorted() on a parallel stream may be slower than sequential because of merge overhead. findFirst() is also costly in parallel — use findAny() if order doesn't matter.
Q: Why does parallelStream().forEach(list::add) produce wrong results?
A: ArrayList is not thread-safe. Multiple threads race on: (1) internal array resize, (2) index increment, (3) element writes. This causes lost writes, duplicate writes, and ArrayIndexOutOfBoundsException. The fix: use .collect(Collectors.toList()) which is designed for parallel collection. It uses merge-friendly collectors that combine partial results safely.
Q: When is a parallel stream actually faster?
A: Parallel streams help when: (1) the source is efficiently splittable (arrays, IntStream ranges, HashSet — NOT LinkedList, Stream.iterate), (2) the pipeline has many elements (N > 10,000 is a rough rule), (3) operations are CPU-intensive (complex math, object creation), and (4) the result combination is cheap. They hurt when: operations are trivial, source splitting is expensive, or there's shared mutable state.
Q: Can map() with side effects cause bugs in streams?
A: Yes — and it's a common anti-pattern. map() is lazy and intermediate. If you put a database update inside map(), it only runs when a terminal operation demands it. Adding .limit(10) later means only 10 updates run. Using .findFirst() means only the first element's update runs. If the stream is parallel, updates may run out of order, multiple times, or not at all. Never put side effects in map() or filter().
Servlet Container Internals & Thread Safety
Single Instance, Multiple Threads
The servlet container creates exactly ONE instance of your servlet (unless you implement SingleThreadModel, which is deprecated). All requests share this instance, with each request handled by a different thread calling service(). This means instance variables are NOT thread-safe unless synchronized. Use request and session attributes for request-scoped data. For application-scoped data, use ServletContext with proper synchronization or concurrent collections.
Servlet 3.0+ Async Support
HttpServletRequest.startAsync() releases the container thread while long-running work continues in a background thread. The response is completed later via AsyncContext.complete(). This is the precursor to reactive programming in Spring WebFlux. Without async, a thread is held for the entire request duration — with 200 threads and 1000 concurrent long-polling requests, 800 requests queue or fail.
Q: Why is init() called only once but service() many times?
A: init() is for expensive one-time setup: loading config, initializing connection pools, warming caches. The servlet spec guarantees exactly one init() call before any service(). service() is the request handler — it must be fast and stateless. The container manages the servlet lifecycle to amortize setup cost across thousands of requests.
Q: What's the difference between ServletContext and ServletConfig?
A: ServletConfig is per-servlet initialization parameters (defined in web.xml or @WebInitParam). ServletContext is shared across the entire web application — all servlets, filters, and listeners can access it. Use ServletConfig for servlet-specific settings (e.g., this servlet's API key). Use ServletContext for app-wide resources (e.g., database connection pool, cache manager).
Q: How does Spring MVC relate to Servlets?
A: Spring MVC is built on the Servlet API. DispatcherServlet is a single servlet that receives all requests and delegates to @Controller methods. It uses HandlerMapping to find the right controller, HandlerAdapter to invoke it, and ViewResolver to render the response. Understanding Servlets is essential because Spring's abstractions leak: thread-safety rules, async support, and filter chains all map directly to Servlet concepts.
Hibernate Session & Transaction Internals
Session == Persistence Context
A Hibernate Session is both a database connection wrapper AND a first-level cache (persistence context). When you call session.get(), Hibernate first checks the persistence context. If the entity is there, no SQL is fired. All entities loaded in a session are tracked for dirty checking — Hibernate compares field values at flush time to generate UPDATE statements. This is why long sessions hurt performance: more entities to track, more memory, slower flushes.
Flush Modes
AUTO (default): flush before every query to ensure query results include pending changes. COMMIT: flush only at transaction commit — faster but queries may see stale data. MANUAL: you control flushing explicitly. ALWAYS: flush before every query (like AUTO but more aggressive). For read-heavy transactions, use COMMIT to reduce unnecessary flushes.
Q: What's the difference between get() and load()?
A: get() returns the entity immediately (or null if not found), hitting the database if not in cache. load() returns a proxy without hitting the database — the actual data is loaded only when you access a field. If the entity doesn't exist, load() throws ObjectNotFoundException on field access, not on the load() call itself. Use load() when you only need the entity's ID (e.g., for setting a foreign key relationship).
Q: Why does LazyInitializationException happen?
A: Hibernate proxies (lazy-loaded entities/collections) need an open Session to load data. If the Session is closed (transaction ended) and you access a lazy field, Hibernate can't fetch the data — there's no database connection. Solutions: (1) keep the Session open (Open Session in View pattern), (2) fetch eagerly with JOIN FETCH or @EntityGraph, (3) initialize proxies within the transaction with Hibernate.initialize(), or (4) use DTO projections to avoid entity traversal entirely.
Q: What's the difference between merge() and saveOrUpdate()?
A: merge() (JPA standard): copies the state of a detached entity to a managed instance and returns the managed copy. The original entity remains detached. saveOrUpdate() (Hibernate-specific): if the entity is transient, saves it; if detached, reattaches it. merge() is safer for detached entities because it doesn't modify the original. In Spring Data JPA, save() calls merge() for detached entities and persist() for new ones.
N+1 Problem & Caching Strategies
N+1: The Silent Performance Killer
With FetchType.LAZY (JPA default), loading N parent entities and accessing their children triggers 1 query for parents + N queries for children = N+1 total queries. For 200 departments with employees, that's 201 queries instead of 1 JOIN. The 1st-level cache helps only if the same child is accessed multiple times in the same Session. In a loop over 200 different parents, every child access fires a new query.
Solution Hierarchy
1. JOIN FETCH: Best when you know at query time that you need the relationship. One query, but may duplicate parent columns. 2. @EntityGraph: Same as JOIN FETCH but declarative — integrates with Spring Data JPA method names. 3. @BatchSize: Fetches children in batches (e.g., 50 at a time). Good when joins cause result set inflation. 4. @Fetch(FetchMode.SUBSELECT): Two queries — one for parents, one subselect for all children. No batch size tuning needed. 5. DTO Projection: For reporting, use JPQL constructor expressions or JdbcTemplate — don't hydrate entities you immediately discard.
Q: Why not just use FetchType.EAGER everywhere?
A: EAGER is a global setting — it forces a JOIN on EVERY query that touches the entity, even when you don't need the relationship. A simple findById() on Ticket now loads the Agent, their Department, their Manager, ad infinitum. This is the "cartesian product explosion" problem. LAZY is the right default; use explicit fetch strategies only when you know you need the data.
Q: How does 2nd Level Cache work with concurrent updates?
A: 2nd Level Cache (e.g., Ehcache, Caffeine, Redis) is a process-wide cache across Sessions. Hibernate uses cache concurrency strategies: READ_ONLY (immutable entities, fastest), NONSTRICT_READ_WRITE (occasional updates, small stale window), READ_WRITE (timestamp-based versioning, prevents dirty reads), TRANSACTIONAL (XA transactions, slowest but safest). Choose based on update frequency. Cache invalidation happens at flush time — stale data is possible between flushes.
Q: What's the difference between Query Cache and 2nd Level Cache?
A: 2nd Level Cache stores entities by primary key — session.get(Employee.class, 5L) hits the cache. Query Cache stores the RESULT SET of a query keyed by the SQL string + parameters — SELECT e FROM Employee e WHERE e.dept = 'IT' hits the cache. Query cache is less effective because: (1) any change to Employee invalidates ALL cached queries touching Employee, (2) it stores IDs, not full entities — still needs 2nd level cache lookups. Use query cache only for truly static reference data.

💡 Bean Scopes: Deep Insight

Why Singleton is Default
Spring chose singleton as default because most services are stateless. Creating new instances for every injection would be wasteful. But this means NEVER store mutable state in a singleton — use ThreadLocal or request-scoped beans for per-request state.
The Proxy Magic
When a request-scoped bean is injected into a singleton, Spring creates a CGLIB proxy. The proxy doesn't contain the real bean — it delegates to the current request's instance via ObjectFactory.getObject(). This is why @Async breaks: the proxy tries to resolve the scope in a new thread where no request exists.
WebSocket Scope
Spring also provides @Scope("websocket") for WebSocket sessions. Each WebSocket connection gets its own bean instance — perfect for chat rooms or real-time games where each connection maintains its own state.

💡 Bean Lifecycle: Deep Insight

@PostConstruct vs InitializingBean
@PostConstruct (JSR-250) runs BEFORE InitializingBean.afterPropertiesSet(). Prefer @PostConstruct because it's framework-agnostic. InitializingBean couples your code to Spring. The order is: constructor → @Autowired → @PostConstruct → afterPropertiesSet() → init-method.
BeanPostProcessor Power
BeanPostProcessor is how Spring implements AOP, @Autowired, @Transactional, and @Async. Every bean passes through ALL registered post-processors. Spring's own AutowiredAnnotationBeanPostProcessor handles @Autowired injection. Custom post-processors can wrap beans, add interceptors, or validate state.
Startup Failure Strategy
If @PostConstruct throws, Spring calls DisposableBean.destroy() on already-created beans in reverse order. This ensures resources are cleaned up. But if cleanup also fails, the original exception is lost — log carefully in both init and destroy methods.

💡 Spring Events: Deep Insight

Event Bus vs Direct Calls
Events decouple publishers from subscribers, but they also hide dependencies. A developer reading orderService.createOrder() can't see that it triggers inventory updates, emails, and audits. Document your events in architecture decision records (ADRs) and use event naming conventions like OrderPlacedEvent (past tense = already happened).
TransactionalEventListener Pitfall
@TransactionalEventListener(AFTER_COMMIT) runs AFTER the transaction commits. If the listener fails, the transaction is already committed — you can't roll it back. For critical operations (like sending a confirmation email), implement a retry mechanism or use an outbox pattern with a persistent message queue.
Spring Modulith
Spring Modulith (new in Spring Boot 3.3+) provides an EventPublicationRegistry that persists events to a database table. If the application crashes after commit but before the listener runs, the event is replayed on restart. This solves the "lost event" problem for transactional events.

💡 Spring MVC Architecture: Deep Insight

HandlerMapping Priority
Spring registers multiple HandlerMapping beans: RequestMappingHandlerMapping (for @RequestMapping), BeanNameUrlHandlerMapping (for bean names as URLs), and SimpleUrlHandlerMapping (for static resources). They have an @Order — RequestMappingHandlerMapping has highest priority (0). If two mappings match, the one with lower order wins.
Content Negotiation
Spring MVC uses ContentNegotiationManager to decide the response format. It checks: (1) URL suffix (.json, .xml), (2) Accept header, (3) request parameter (?format=json). For REST APIs, configure ContentNegotiationConfigurer.favorPathExtension(false) to disable URL suffix matching — it's a security risk (path traversal via .json).
Async Request Processing
Use Callable<T> or DeferredResult<T> to free the servlet thread while waiting for long operations. With Callable, Spring uses a TaskExecutor. With DeferredResult, YOU set the result from any thread. This is how Spring WebFlux's reactive model works under the hood — but with Servlet API compatibility.

💡 @Transactional Propagation: Deep Insight

Self-Invocation Problem
If OrderService.placeOrder() (REQUIRED) calls this.audit() (REQUIRES_NEW), the REQUIRES_NEW is IGNORED. Why? Because this.audit() bypasses the Spring proxy. The solution: inject OrderService into itself (@Autowired private OrderService self;) or use AopContext.currentProxy().
Connection Pool Exhaustion
REQUIRES_NEW suspends the current connection and gets a NEW one from the pool. If you nest 5 REQUIRES_NEW calls, you need 6 connections. With a pool of 20 and 4 concurrent requests, you're out of connections. Monitor HikariPool-1 - Active connections and set leakDetectionThreshold to catch this.
Read-Only Optimization
@Transactional(readOnly = true) is a HINT, not enforcement. Hibernate uses it to: (1) Skip dirty checking (no snapshot comparison), (2) Disable automatic flushing, (3) Use read-only JDBC connections if the driver supports it. For read-heavy services, this can improve throughput by 15-30%.

💡 Actuator & Micrometer: Deep Insight

The Volkswagen Leak
In 2024, researchers discovered a Volkswagen telematics service with an exposed /actuator/heapdump endpoint. The heapdump contained AWS credentials, API keys, and 9TB of customer GPS data. The fix: (1) Expose only health+prometheus, (2) Put Actuator on a separate management port, (3) Require mTLS from monitoring systems, (4) Use Kubernetes NetworkPolicies to restrict ingress.
Micrometer vs Dropwizard Metrics
Micrometer is a facade (like SLF4J for metrics). It supports Prometheus, Graphite, Datadog, CloudWatch, and 15+ backends. Use Counter for monotonically increasing values, Timer for durations (auto-calculates percentiles), Gauge for current values, and DistributionSummary for arbitrary distributions. NEVER use user IDs or request IDs as tags — it creates cardinality explosion.
Custom Metrics Best Practices
Use @Timed and @Counted on service methods, not controllers (controllers are auto-instrumented). Add consistent tags: application, environment, team. Set SLOs (Service Level Objectives) with management.metrics.distribution.slo to get histogram buckets. Export to Prometheus with /actuator/prometheus and alert on p99 latency exceeding your SLO.

💡 Custom Auto-Configuration: Deep Insight

Auto-Config Ordering
Use @AutoConfigureAfter(DataSourceAutoConfiguration.class) when your auto-config needs a DataSource bean. Use @AutoConfigureBefore(WebMvcAutoConfiguration.class) to set up custom converters before Spring MVC configures its defaults. The order matters because beans created earlier can be referenced by later auto-configs.
spring.factories vs imports
Spring Boot 2.7+ uses META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (one class per line). The old META-INF/spring.factories still works but is deprecated. For Spring Boot 3.x, use the new file. Also, auto-config classes should be in a package OUTSIDE the component scan path — typically com.yourcompany.autoconfigure while the main app is in com.yourcompany.app.
Testing Auto-Configuration
Use @AutoConfigureTestDatabase and TestAutoConfigurationPackage to test auto-configs. ApplicationContextRunner is the best tool: new ApplicationContextRunner().withUserConfiguration(MyAutoConfig.class).run(context -> assertThat(context).hasSingleBean(MyService.class));. Test both positive (conditions met) and negative (conditions not met) cases.

💡 GraalVM Native Images: Deep Insight

AOT vs JIT Trade-offs
JVM uses JIT (Just-In-Time) compilation: starts fast, optimizes hot code over time. GraalVM uses AOT (Ahead-of-Time): slow build, but peak performance from the first request. For long-running services (>1 hour), JIT often wins because it profiles and optimizes based on real workload. For serverless (<5 min), native images win because there's no warm-up time.
Reflection Configuration
GraalVM needs explicit reflection config because it can't analyze dynamic class loading at build time. Spring Boot 3.x generates most config automatically, but you must hint: (1) Classes deserialized with Jackson, (2) JPA entities with complex mappings, (3) Custom proxy interfaces, (4) Resource bundles. Use @RegisterReflectionForBinding(MyDto.class) as a shortcut.
Native Image Debugging
Native images don't support standard JVM debugging (no JVMTI). Use: (1) --trace-class-initialization to find classes initialized at build time vs run time, (2) --report-unsupported-elements-at-runtime to defer errors, (3) GDB for C-level debugging, (4) Spring Boot's native:compile with -Dspring.aot.debug=true for AOT processing logs. Profile in JVM mode first, then port.
